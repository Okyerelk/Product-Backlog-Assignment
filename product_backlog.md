# Initial Product Backlog – VCU Summer 2026 Course Registration System

## Key Stakeholders and Users
- Students: Search summer courses, view session details, and enroll in sessions.
- Instructors: View assigned teaching sessions and view student rosters for sections they teach.
- Registrar/Admin (authorized users): Support registration by viewing enrollment information and ensuring registration rules are enforced.

---

## Product Backlog User Stories (Initial)

### User Story 1
**As a student, I want to search summer courses by department, instructor, or course number so that I can find classes that match what I want to take.**  
**Priority:** High  
**Priority justification:** Search is essential for students to find courses and it is a dependency for enrollment since students must locate sessions before they can register.

### User Story 2
**As a student, I want to view session details (course number, section number, modality, instructor, and maximum seats) so that I can choose the right section for my needs.**  
**Priority:** High  
**Priority justification:** Students need accurate session details to make enrollment decisions, and this directly supports the primary registration workflow.

### User Story 3
**As a student, I want to enroll in a session so that I can register for summer courses.**  
**Priority:** High  
**Priority justification:** Enrollment is the core purpose of the system and delivers the most stakeholder value because it enables actual registration.

### User Story 4
**As the system, I want to prevent a student from registering for two sessions of the same course so that registration rules are enforced correctly.**  
**Priority:** Medium  
**Priority justification:** This rule protects data integrity and prevents invalid schedules, and it becomes critical once basic enrollment exists.

### User Story 5
**As the system, I want to block enrollment when a session reaches maximum capacity so that sections do not exceed their allowed seats.**  
**Priority:** Medium  
**Priority justification:** Capacity enforcement prevents over-enrollment and reduces manual fixes by staff, and it directly supports a realistic registration process.

### User Story 6
**As a student, I want to view the sessions I am currently enrolled in so that I can confirm my summer schedule.**  
**Priority:** Medium  
**Priority justification:** This supports student confidence and reduces support requests, and it depends on enrollment data being stored.

### User Story 7
**As an instructor, I want to view the sessions I am assigned to teach so that I can plan my summer teaching schedule.**  
**Priority:** Medium  
**Priority justification:** Instructors are key stakeholders, and this provides strong value while relying on session and instructor assignment data already being available.

### User Story 8
**As an instructor, I want to view a roster of enrolled students for a session so that I can prepare course communication and manage logistics.**  
**Priority:** Medium  
**Priority justification:** Rosters are important for instruction and administration, and this story depends on enrollments being recorded.

### User Story 9
**As an authorized user, I want to look up the sessions a specific student is enrolled in so that I can support advising and troubleshoot registration issues.**  
**Priority:** Medium  
**Priority justification:** This improves efficiency for support staff and advising, and it depends on enrollment data being available.

### User Story 10
**As a student, I want to drop a session so that I can adjust my schedule if my plans change.**  
**Priority:** Low  
**Priority justification:** Dropping is useful but not required to demonstrate initial registration functionality, and it can be added after the core enroll workflow is stable.

### User Story 11
**As a student, I want to view prerequisite information for a course so that I can avoid registering for a course I am not prepared for.**  
**Priority:** Low  
**Priority justification:** Prerequisites improve decision-making, but they are less time-sensitive than search and enrollment and may require extra data and rule handling.
